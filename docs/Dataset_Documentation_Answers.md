# Dataset Documentation Answers (Questions 1 to 9)

Here are the formatted answers corresponding to the sections in your "Dataset Documentation Template.docx" based on the `ApexHunter 2.0` project. 

### 1. Basic dataset information
- **Dataset name:** Formula 1 Telemetry Dataset (ApexHunter Project)
- **Domain / application area:** Sports Analytics / Motorsports / Automotives / Computer Vision
- **Type of data:** Structured (Parquet files) and Unstructured (MP4 Video, Onboard Camera Frames, Satellite/Track Map Images)
- **Source of dataset:** Official F1 Timing and Telemetry APIs (via `fastf1` Python Library), YouTube (via `yt-dlp`), and Track Maps/Satellite Images.
- **Dataset link / reference:** `https://fastf1.readthedocs.io/` 
- **License / usage rights:** Non-commercial use (F1 data is proprietary but permitted for non-commercial/academic analysis via the FastF1 community tool).

### 2. Data characteristics
**For Structured Data:**
- **Number of records (rows):** ~57,874,119 rows (Cleaned Dataset)
- **Number of features (columns):** 10 core features in the clean dataset
- **Attribute names and description:** 
  - `Speed`: Car speed in km/h
  - `RPM`: Engine revolutions per minute
  - `Throttle`: Throttle pedal depression percentage
  - `Brake`: Brake pedal depression percentage
  - `X`, `Y`: Spatial coordinates of the car on the track (Used to build the state-space grid for optimal racing line search algorithms like A*, BFS, Dijkstra)
  - `Time`, `SessionTime`: Timestamps relative to lap/session start
  - `nGear`: Current gearbox gear
  - `Driver`: Three-letter identifier of the driver
- **Data types (numeric, categorical, etc.):** Sub-types are `float32` (Speed, RPM, Throttle, Brake, X, Y), string/categorical (Driver), and temporal/timedelta (Time).
- **Target variable (if applicable):** X, Y coordinates with Speed/Brake weights serve as the grid for search algorithms. For computer vision, the semantic segmentation masks define the target (Road, Curb, Background).

**For Unstructured Data:**
- **Total number of files:** ~48 files (Track Map Images + 2023 & 2024 Pole Lap Videos)
- **File format:** MP4 (1080p Video Files), PNG/JPEG (Satellite Images for UI Background)
- **Total dataset size (MB/GB):** 
  - Structured data: ~1691 MB (~1.7 GB) of cleaned Parquet files and ~1720 MB of raw files.
  - Unstructured data: ~1.8 GB (~40-50 MB per Pole Lap Video from all ~44 qualifying sessions across the 2023 and 2024 seasons) + ~10 MB of Satellite/Track Images.
- **Annotation availability:** Semantic Segmentation masks generated by YOLOv11 (Classes: 0=Road, 1=Curb, 2=Background) for computing the "Deviation Metric" from the apex on onboard video.

### 3. Big Data perspective (5 Vs)
- **Volume:** The structured dataset contains ~1.7 GB of data distributed across 92 compressed Parquet files with nearly 58 million rows of high-frequency records. The unstructured video dataset encompasses another ~1.8 GB of 1080p MP4 onboard camera footage from all 2023 and 2024 pole position laps to train the YOLOv11 Computer Vision model.
- **Velocity:** Telemetry is generated at a very high frequency (e.g., millions of records per race). The ingestion pipeline processes bulk batch downloads from seasonal racing sessions.
- **Variety:** The dataset incorporates structured tabular car physics data (`.parquet`), unstructured multimedia track footage (`.mp4` onboard pole laps from 2023/2024), and Satellite/Track visual images.
- **Veracity:** High source credibility (Official F1 APIs), but raw data inherently contains "dirty" signals. Furthermore, the YOLOv11 framework ensures robustness to lighting changes, motion blur, and occlusions in the video data compared to traditional OpenCV methods.
- **Value:** The dataset empowers an interactive dashboard, allows search algorithms to find the optimal racing line via the X/Y telemetry grid, and uses computer vision (Semantic Segmentation) to generate a "Deviation Metric" to determine if drivers hit the apex, visualized via an Augmented Reality HUD overlay.

### 4. Data quality assessment
- **Missing values analysis:** Raw telemetry often exhibits intermittent gaps due to dropped car-to-pitwall network packets.
- **Duplicate records:** Negligible/None. Structured around precise chronological timeticks per driver.
- **Outliers:** Erroneous sensor spikes occasionally report impossible speeds (>380 km/h) or engine revolutions (>15000 RPM limit).
- **Noise / inconsistencies:** GPS coordinates (`X`, `Y`) may drift slightly from actual track bounds due to signal noise.
- **Data imbalance:** Not directly applicable as a class imbalance, but completed race laps outnumber qualifying laps significantly, giving more volume to race-pace data.
- **Bias considerations:** The data captures cars built for maximum performance on perfectly paved circuits. Drivers inherently reflect distinct tiers of car performance (e.g., Red Bull vs. Haas).

### 5. Data preprocessing details
- **Data cleaning techniques used:** Dropping entirely corrupted files or rows missing critical core columns (`Speed`, `RPM`, `X`, `Y`).
- **Handling missing values:** Applied forward-filling (`ffill()`) for numeric arrays to bridge small temporal gaps left by dropped micro-packets during a lap.
- **Encoding techniques:** String placeholders are used for the `Driver` identifier; if completely missing, designated as `UNKNOWN`.
- **Feature scaling / normalization:** Domain-specific outlier clipping: `Speed` capped at 380 km/h, `RPM` capped at 15,000, and `Throttle/Brake` strictly bounded between 0 and 100.
- **Transformation steps:** Downcasted 64-bit floats to 32-bit `float32` for dataset optimization. For Computer Vision: Extracting a road mask contour and calculating the Euclidean Distance (pixels) between the "Car's Nose" pixel and the "Road Edge" to compute the Deviation Metric. Image Blending logic for AR UI effects (HUD).

### 6. Exploratory Data Analysis (EDA)
- **Summary statistics:** Analyzing median cornering speeds across various tracks, and the threshold of maximum G-force impacts on straightaways.
- **Visualization techniques used:** Interactive full-length line graphs with dynamic time-scrubbers, spatial scatter mapping overlaid on Satellite Images, and Image Blending (Frame = 0.7 * Original + 0.3 * Mask) on video frames for Augmented Reality HUD.
- **Correlation analysis:** Direct correlation observed between longitudinal track position and throttle vs. brake pressure applications.
- **Key observations:** Brake traces sharply increase precisely before sharp hairpin turns, corresponding identically to a drastic descent in `RPM` and `Speed` graphs.

### 7. Tools and technologies used
- **Programming language(s):** Python 3.10+
- **Libraries / frameworks:** `pandas`, `numpy`, Streamlit (for UI layout)
- **Analytics & CV Frameworks:** YOLOv11 (YOLOv11-Seg for Semantic Segmentation)
- **Visualization tools:** Plotly Express / Plotly Graph Objects, OpenCV (for Image Blending / HUD projection)
- **Big data tools (if any):** Apache Parquet (columnar storage engine enabling scalable file I/O operations).
- **Any other tool / technology:** `yt-dlp` (for extracting auxiliary unstructured video), `fastf1` wrapper.

### 8. Ethical and legal considerations
- **Personal data involved:** None. The data pertains entirely to public sporting figures during a professional broadcast.
- **Consent obtained:** The data is broadcast publicly and retrieved using an explicitly open-source community API wrapper for educational/non-commercial intents.
- **Anonymization techniques used:** Not required.
- **Compliance with data protection norms:** Fully compliant as no Personally Identifiable Information (PII) beyond public domain driver acronyms is exposed.
- **Sensitive Attributes if any:** None.

### 9. Data storage and management
- **Storage type:** Local Disk (Simulated Data Lake Structure)
- **Storage platform used:** Hierarchical Local File System Storage.
- **Data format stored:** Snappy-compressed Apache Parquet Files `.parquet`, `.mp4` video files, and `.png`/`.jpg` satellite images.
- **Data organization structure:** Partitioned logically through directory layering: `data_lake/season_data/` (Raw Source Zone) and `data_lake/clean_data/` (Curated / Processed Zone).
- **Backup and recovery strategy:** Data is stateless on the repository and can easily be retrieved again via the automated ingestion script (`download_season_data.py`).
- **Security measures:** OS-level access control. Total footprint is localized on the machine.
- **Scalability considerations:** Leveraging Parquet ensures the system can seamlessly read gigabytes of data into pandas DataFrames in seconds, easily supporting data from future seasons without massive resource upgrades.
- **Ingestion method:** Automated Python scripts executing controlled API calls with internal caching schemas (`cache/`) to avoid repeatedly hitting external servers.
